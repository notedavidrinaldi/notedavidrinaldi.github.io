PiTUSK untuk Tim Non-Teknis

Versi 1 halaman untuk operasi upload harian tanpa tenggelam di detail implementasi

PiTUSK itu apa?

PiTUSK adalah petugas jalur upload untuk file yang dikirim dari perangkat atau sistem edge.

Kalau koneksi putus di tengah jalan, upload tidak perlu diulang dari nol. PiTUSK menyimpan progresnya lalu melanjutkan dari posisi terakhir.

  • Create: buka sesi upload.
  • Patch: kirim file per bagian.
  • Head: cek posisi terakhir.
  • Finalize: selesaikan upload dan pindahkan ke lokasi akhir.

Kenapa penting untuk operasi

  • Tidak membuang waktu mengulang file besar dari awal.
  • Lebih aman untuk jaringan edge yang tidak selalu stabil.
  • Status sesi bisa dilihat jelas: baru, sedang upload, selesai, sinkron, atau gagal.
  • Mudah dibedakan apakah masalah ada di jaringan, token, offset, atau sinkronisasi akhir.

Status yang perlu dipahami

  • new: sesi baru dibuat, file belum mulai dikirim.
  • uploading: chunk sedang masuk.
  • finalized: file sudah lengkap dan dipindahkan ke lokasi final.
  • synced: file sudah diteruskan ke target sinkronisasi.
  • failed: ada gangguan yang perlu dicek operator atau engineer.

Peta mental kerja

  1. Buat sesi upload.
  2. Kirim file per chunk.
  3. Jika koneksi putus, cek offset terakhir.
  4. Lanjutkan dari offset itu.
  5. Finalize jika total file sudah lengkap.

Cara kerja tim harian

  • Mulai shift: cek /healthz dan /readyz.
  • Saat upload macet: cek offset terakhir dengan HEAD.
  • Jika file belum muncul di lokasi akhir: cek apakah sesi masih uploading atau belum finalize.
  • Jika sinkronisasi lambat: tandai sebagai isu reliability, bukan langsung dianggap file hilang.
  • Jika auth gagal: verifikasi token API lebih dulu sebelum investigasi storage.

Command kunci (tim operasi)

Cek service hidup

curl http://localhost:8080/healthz

Memastikan proses layanan aktif.

Cek service siap

curl http://localhost:8080/readyz

Memastikan layanan siap menerima upload baru.

Lanjutkan upload

curl -I http://localhost:8080/v1/uploads/<upload-id>

Mengambil offset terakhir agar upload bisa dilanjutkan.

Cek status sesi

curl http://localhost:8080/v1/uploads/<upload-id>/status

Melihat status yang sedang berjalan pada sesi itu.

Finalize file

curl -X POST http://localhost:8080/v1/uploads/<upload-id>/finalize

Menyelesaikan upload setelah seluruh chunk masuk.

Lanjut kerja dev

make resume
make resume-json

Untuk meneruskan roadmap pengembangan yang sempat terputus.

Kapan perlu eskalasi

  • Offset tidak bergerak meski chunk baru sudah dikirim.
  • Status berulang kali berhenti di failed.
  • Service sehat tetapi file tidak pernah sampai ke lokasi final.
  • Auth failure muncul berulang walau token sudah benar.
  • Banyak sesi upload tertinggal dan tidak terselesaikan.
