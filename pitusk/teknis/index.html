Arsitektur

PiTUSK berjalan sebagai service upload edge: API layer → upload engine → local staging/storage → metadata SQLite → sync worker.
KomponenPeranCatatan
API LayerHTTP endpointValidasi auth, header, routing, dan error mapping.
Upload EngineLifecyle sessionCreate, write chunk, offset validation, finalize, delete.
Storage LocalStaging + final fileMemisahkan lokasi upload sementara dan lokasi hasil akhir.
Metadata RepoStatus persistenSQLite menyimpan offset, status, dan info sesi.
Sync WorkerDistribusi lanjutanTarget lokal tersedia, adapter lain masih berkembang.

Alur Upload

Create session → PATCH chunk → Update offset
              → HEAD for resume → Finalize
              → Sync worker (optional)

Alur ini menjaga proses tetap idempotent secara operasional: client cukup tahu offset terakhir, lalu melanjutkan dari sana.

Mulai Cepat (5 menit)

  1. Jalankan service
    cd /Users/davidrinaldi/Documents/PROJECT-DAVID/PiTUSK
go run ./cmd/piresum
  2. Buat sesi upload
    curl -X POST http://localhost:8080/v1/uploads \
  -H "Content-Type: application/json" \
  -d '{"filename":"video.mp4","size_total":2048}'
  3. Kirim chunk
    curl -X PATCH http://localhost:8080/v1/uploads/<upload-id> \
  -H "Upload-Offset: 0" \
  --data-binary @chunk.bin
  4. Resume jika putus
    curl -I http://localhost:8080/v1/uploads/<upload-id>
  5. Finalize
    curl -X POST http://localhost:8080/v1/uploads/<upload-id>/finalize

Kontrak API

MethodEndpointFungsi
POST/v1/uploadsMembuat sesi upload baru.
PATCH/v1/uploads/{id}Mengirim chunk berdasarkan Upload-Offset.
HEAD/v1/uploads/{id}Mengambil offset terkini untuk resume.
GET/v1/uploads/{id}/statusMengambil status sesi.
POST/v1/uploads/{id}/finalizeMenyelesaikan dan memindahkan file ke jalur final.
DELETE/v1/uploads/{id}Membatalkan sesi upload.
GET/healthzHealth check.
GET/readyzReadiness check.

Mode Framework

import "github.com/notedavidrinaldi/PiTUSK/pkg/pitusk/v1"

cfg, err := v1.LoadConfig()
server, err := v1.New(cfg, nil)
if err != nil { panic(err) }
if err := server.Run(); err != nil { panic(err) }

Jalur publik yang direkomendasikan sekarang adalah pkg/pitusk/v1. Layer lama pkg/pitusk masih dipertahankan untuk backward compatibility.

Konfigurasi Dasar

  • PI_TUSK_ADDR default :8080
  • PI_TUSK_STAGING_DIR default data/staging
  • PI_TUSK_FINAL_DIR default data/files
  • PI_TUSK_DB_PATH default data/piresum.sqlite
  • PI_TUSK_CHUNK_SIZE, PI_TUSK_MAX_FILE_SIZE, PI_TUSK_MAX_CONCURRENT
  • PI_TUSK_SYNC_ENABLED, PI_TUSK_SYNC_INTERVAL, PI_TUSK_SHUTDOWN_TIMEOUT, PI_TUSK_API_TOKEN

Reliability

  • Session dan offset dipersistenkan ke SQLite.
  • Worker sinkronisasi dasar sudah tersedia.
  • Target local copy sudah tersedia sebagai adapter awal.
  • Backlog reliability: adapter S3/MinIO, retry policy, DLQ, dan idempotency sync.

Keamanan

  • Auth token middleware sudah tersedia.
  • Validasi ukuran file, offset, dan status sesi sudah berjalan.
  • Prioritas berikutnya: sanitize nama file, proteksi path traversal, rate limit, dan audit log event sensitif.

Operasional

curl http://localhost:8080/healthz
curl http://localhost:8080/readyz
make resume
make resume-json

Health dan readiness cocok untuk supervisi service, sedangkan command resume dipakai untuk melanjutkan pengembangan yang terhenti.

Roadmap

  • Foundation dan core framework contract sudah tersedia.
  • Sync reliability masih aktif dikembangkan.
  • Security hardening dan observability menjadi agenda sebelum rilis mayor.
  • Packaging komunitas disiapkan sebagai fase lanjutan.
