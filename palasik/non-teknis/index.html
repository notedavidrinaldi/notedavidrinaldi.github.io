PALASIK untuk Tim Non-Teknis

Versi 1 halaman untuk operasi harian tanpa detail implementasi

PALASIK itu apa?

PALASIK adalah petugas gerbang keamanan untuk data sensor/IoT.

Setiap data yang masuk ke gateway tidak langsung dianggap aman. Setiap event ditentukan aksinya:

  • ALLOW: diproses
  • DENY: ditolak
  • MONITOR / RESTRICT / CHALLENGE: dipantau khusus

Kenapa penting untuk operasi

  • Keputusan konsisten karena berbasis aturan, bukan ad hoc.
  • Ada bukti keputusan berupa reason code dan log audit.
  • Bisa rollback cepat jika policy baru membuat sistem tidak stabil.
  • Indikator harian tersedia: deny spike, trust turun, dan status pipeline.

Peta mental kerja

  1. Event masuk dari sensor/gateway.
  2. Sistem menilai tingkat trust.
  3. Policy engine memilih keputusan.
  4. Keputusan dieksekusi + tercatat.

Cara kerja tim harian

  • Mulai shift: jalankan palasik check --config config.yaml lalu palasik status --config config.yaml.
  • Sebelum deploy policy baru: ambil snapshot + smoke gate policy.
  • Jika ada anomali: rollback cepat ke snapshot terakhir yang valid.
  • Setelah normal: lanjut monitoring rutin dan pencatatan evidence.

Command kunci (tim operasi)

Validasi sehat

palasik check --config config.yaml

Menguji startup pipeline trust dan policy.

Pemantauan 5 menit

palasik status --config config.yaml

Melihat metrik dan alert terkini.

Guard pre-deploy

palasik policy-snapshot --config config.yaml
palasik policy-deploy-check --config config.yaml --require-allow

Wajib sebelum perubahan policy ke produksi.

Rollback cepat

palasik policy-rollback \
  --config config.yaml \
  --snapshot runs/policy_snapshots/<file>.snapshot.yaml

Dipakai jika terjadi deny spike / latensi anomali.

Kapan rollback dilakukan

  • Penalty DENY melonjak tajam dan tidak wajar.
  • Latency naik drastis setelah perubahan policy.
  • Keputusan tidak sesuai kondisi bisnis saat ini.

Prosesnya cepat: ambil status saat ini, rollback ke snapshot valid, lalu jalankan check ulang.

Catatan akhir

Baseline PALASIK disetel zero trust by default. Keamanan diprioritaskan dulu, lalu menjaga layanan tetap jalan.
