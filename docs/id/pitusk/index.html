Konsep Inti

PiTUSK menangani upload file besar secara bertahap tanpa mengorbankan kemampuan resume saat koneksi terputus.

  • Sesi upload disimpan sebagai metadata persisten di SQLite.
  • Chunk ditulis ke staging file dengan validasi offset.
  • File difinalisasi ke jalur akhir setelah seluruh data lengkap.
  • Worker sinkronisasi dapat meneruskan file final ke target lain.

Posisi PiTUSK di Rumah Aplikasi

Jika PALASIK berfokus pada keputusan keamanan event IoT, maka PiTUSK berfokus pada ketahanan transfer file di lingkungan edge yang terbatas.

PiTUSK sebagai service

Dipakai langsung melalui endpoint HTTP untuk create session, upload chunk, resume, dan finalize.

PiTUSK sebagai framework

Dipakai sebagai package publik pkg/pitusk/v1 agar bisa ditanamkan ke project lain.

Kontrak API

Contoh Flow Client

# create session
curl -X POST http://localhost:8080/v1/uploads \
  -H "Content-Type: application/json" \
  -d '{"filename":"sample.bin","size_total":12345}'

# upload chunk
curl -X PATCH "http://localhost:8080/v1/uploads/<upload-id>" \
  -H "Upload-Offset: 0" \
  --data-binary @chunk.bin

# check offset after interrupt
curl -I http://localhost:8080/v1/uploads/<upload-id>

# finalize
curl -X POST http://localhost:8080/v1/uploads/<upload-id>/finalize

Mode Framework

PiTUSK tidak hanya diposisikan sebagai binary. Project ini juga disusun agar bisa dipakai sebagai framework upload inti.

  • API publik utama berada di github.com/notedavidrinaldi/PiTUSK/pkg/pitusk/v1.
  • Layer lama pkg/pitusk tetap hidup untuk backward compatibility.
  • Konfigurasi dan inisialisasi server sudah dikemas agar bootstrap lebih konsisten.
Inti arahnya: aplikasi dan library berbagi kontrak yang sama, sehingga tim bisa mulai dari binary lalu naik kelas ke integrasi framework tanpa ganti konsep.

Arsitektur Komponen

API Layer

Validasi auth, parsing header, middleware, dan routing.

Upload Engine

Lifecycle sesi upload, kontrol offset, status, dan concurrency.

Storage Layer

Staging file lokal, finalization path, dan repository metadata SQLite.

Sync Engine

Worker background untuk copy atau sinkronisasi ke target lain.

Config + Runtime

Bootstrap aplikasi, lifecycle, dan graceful shutdown.

Operations

Health, readiness, logging, serta cleanup sesi gagal di tahap berikutnya.

Data Flow

  1. Client membuat sesi upload via POST.
  2. Client mengirim chunk via PATCH.
  3. Upload engine menulis chunk lalu memperbarui offset.
  4. Jika koneksi putus, client mengambil offset terkini via HEAD.
  5. Setelah upload penuh, client memanggil finalize.
  6. Sync worker meneruskan file ke target sinkronisasi bila diaktifkan.

Operasional dan Konfigurasi

Environment variable utama yang paling sering dipakai:

  • PI_TUSK_ADDR dengan default :8080.
  • PI_TUSK_STAGING_DIR untuk area staging file.
  • PI_TUSK_FINAL_DIR untuk lokasi file final.
  • PI_TUSK_DB_PATH untuk SQLite metadata.
  • PI_TUSK_CHUNK_SIZE, PI_TUSK_MAX_FILE_SIZE, dan PI_TUSK_MAX_CONCURRENT untuk kontrol upload.
  • PI_TUSK_SYNC_ENABLED dan PI_TUSK_SYNC_INTERVAL untuk worker sinkronisasi.
  • PI_TUSK_SHUTDOWN_TIMEOUT untuk graceful shutdown.
  • PI_TUSK_API_TOKEN untuk bearer token opsional.
Untuk cek hidupGunakan /healthz agar cepat tahu service aktif.
Untuk cek siap trafikGunakan /readyz sebelum menempatkan service di belakang proxy atau supervisor.
Untuk lanjut kerja devmake resume dan make resume-json membaca status roadmap dan action berikutnya.

Keamanan dan Reliability

  • Middleware auth token sudah tersedia.
  • Validasi ukuran file, offset, dan status sesi sudah diterapkan.
  • Worker sinkronisasi dasar dan target lokal sudah tersedia.
  • Backlog prioritas berikutnya meliputi sanitize nama file, proteksi path traversal, retry/backoff, dan audit log event sensitif.

Roadmap Pengembangan

Status garis besarnya saat ini:

  • Tahap 0: foundation selesai.
  • Tahap 1: core framework contract berjalan dan mayoritas komponen utama sudah ada.
  • Tahap 2: sync adapter dan reliability masih berlanjut, terutama untuk S3/MinIO, retry, DLQ, dan idempotency.
  • Tahap 3-5: hardening keamanan, observability, command operasi, dan packaging komunitas masih jadi backlog aktif.
