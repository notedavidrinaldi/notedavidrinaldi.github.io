Rancangan Keputusan Edge untuk Keamanan PALASIK

Penulis: David Rinaldi · Oktober 2025 · Update pendekatan risiko real-time

Abstrak

Pada proyek PALASIK, tantangan utama bukan hanya mendeteksi event, tapi menentukan keputusan cepat saat kondisi perangkat dan link jaringan berubah dalam hitungan detik. Tulisan ini memetakan model trust score yang dipakai di edge agar keputusan bisa diambil tanpa menunggu seluruh pipeline cloud.

Permasalahan

Pengamanan IoT di pabrik sering dibangun dengan pola: data selalu masuk, kemudian dinilai secara reaktif. Saat koneksi terputus, sistem cenderung berhenti bereaksi atau malah mengizinkan event yang belum tervalidasi. Kita butuh model yang menjaga kesinambungan keputusan sambil tetap disiplin terhadap risiko.

Rancangan Model

  1. Trust score dipetakan dari empat sinyal: source, kondisi node, nilai anomali temporal, dan status policy.
  2. Skor di-update terus menerus berdasarkan event terbaru agar kebijakan ALLOW / RESTRICT / DENY / QUARANTINE tidak statis.
  3. Setiap keputusan diekspor sebagai jejak audit supaya dapat ditelusuri saat investigasi berikutnya.
  4. Fallback rule disiapkan untuk mode koneksi buruk, termasuk mode degradasi aman agar traffic kritis tetap bisa diproses.

Hasil Lapangan

Model ini diuji pada simulasi gateway dengan beban event sederhana, lalu dibandingkan dengan pendekatan aturan tetap. Hasil awal yang saya dapat:

  • Respons kebijakan rata-rata menjadi lebih stabil saat burst traffic karena tidak membuat evaluasi ulang berulang ke cloud.
  • Jumlah event yang tertahan saat kondisi risk tinggi menurun karena threshold dinamis mengikuti jam operasi lapangan.
  • Audit log lebih kaya, sehingga korelasi insiden lebih cepat saat review mingguan.

Catatan Implementasi

Di sisi praktis, bagian terpenting bukan hanya logika score itu sendiri, tapi disiplin observabilitas. Tanpa metrik kesehatan modul, perubahan aturan akan terlihat seperti “magic box.”

Kesimpulan Sementara

Kebijakan edge yang adaptif jauh lebih realistis untuk IIoT karena kondisi fisik jaringan tidak pernah ideal. Prinsipnya sederhana: lebih baik keputusan lokal konsisten, cepat, dan audit-able daripada menunggu analisis lengkap di luar edge.

Karyaku ini ditulis dari pengalaman langsung saat meramu desain PALASIK pada versi awal sistem, lalu disederhanakan agar alurnya tetap bisa dipakai untuk dokumentasi internal tim.
